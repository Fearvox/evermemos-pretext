## Pretext API Surface

### Public Exports from `src/layout.ts`

All exports use named ESM export syntax. No default export.

#### Core Functions

```ts
// Fast path — opaque handle, no segment data
prepare(text: string, font: string, options?: PrepareOptions): PreparedText

// Rich path — exposes segment arrays for custom rendering (bidi, line-by-line)
prepareWithSegments(text: string, font: string, options?: PrepareOptions): PreparedTextWithSegments

// Pure arithmetic, ~0.0002ms — call on every resize
layout(prepared: PreparedText, maxWidth: number, lineHeight: number): LayoutResult

// Rich layout with per-line text/width/cursors — off the resize hot path
layoutWithLines(prepared: PreparedTextWithSegments, maxWidth: number, lineHeight: number): LayoutLinesResult

// Batch geometry, no string materialization — use for shrinkwrap, aggregate geometry
walkLineRanges(prepared: PreparedTextWithSegments, maxWidth: number, onLine: (line: LayoutLineRange) => void): number

// Streaming single-line step — for variable-width/obstacle-aware layout
layoutNextLine(prepared: PreparedTextWithSegments, start: LayoutCursor, maxWidth: number): LayoutLine | null

// Cache management
clearCache(): void   // clears analysis + measurement + segmenter caches
setLocale(locale?: string): void  // retargets Intl.Segmenter, calls clearCache()

// Diagnostic only — not for production use
profilePrepare(text: string, font: string, options?: PrepareOptions): PrepareProfile
```

#### Public Types

```ts
// Opaque handle — branded type, no internal fields accessible
type PreparedText = { readonly [preparedTextBrand]: true }

// Rich variant — adds segments: string[] aligned with internal parallel arrays
type PreparedTextWithSegments = PreparedText & {
  segments: string[]  // segment text, e.g. ['hello', ' ', 'world']
  // plus all InternalPreparedText fields (widths, kinds, segLevels, etc.)
}

type LayoutResult = {
  lineCount: number  // number of wrapped lines
  height: number     // lineCount * lineHeight
}

type LayoutLine = {
  text: string         // full text of this line
  width: number        // measured width of this line
  start: LayoutCursor  // inclusive start cursor
  end: LayoutCursor    // exclusive end cursor
}

type LayoutLineRange = {
  width: number
  start: LayoutCursor
  end: LayoutCursor
  // no text — use layoutNextLine() or materialize manually if text is needed
}

type LayoutLinesResult = LayoutResult & {
  lines: LayoutLine[]
}

type LayoutCursor = {
  segmentIndex: number   // index into prepared segments array
  graphemeIndex: number  // grapheme index within segment; 0 at segment boundaries
}

type PrepareOptions = {
  whiteSpace?: WhiteSpaceMode  // 'normal' (default) | 'pre-wrap'
}

type PrepareProfile = {
  analysisMs: number
  measureMs: number
  totalMs: number
  analysisSegments: number
  preparedSegments: number
  breakableSegments: number
}

type PreparedLineChunk = {
  startSegmentIndex: number
  endSegmentIndex: number
  consumedEndSegmentIndex: number
}
```

#### Key Internal Detail: `getMeasureContext()` throws unconditionally on the server

```ts
// From measurement.ts — no graceful fallback, no null return
export function getMeasureContext(): CanvasRenderingContext2D | OffscreenCanvasRenderingContext2D {
  if (typeof OffscreenCanvas !== 'undefined') { ... }
  if (typeof document !== 'undefined') { ... }
  throw new Error('Text measurement requires OffscreenCanvas or a DOM canvas context.')
}
```

This is the hard SSR boundary. Any import path that reaches `prepare()` on the server will throw at request time.

#### `measureContext` is module-level state

```ts
let measureContext: CanvasRenderingContext2D | OffscreenCanvasRenderingContext2D | null = null
```

There is no per-request isolation. A failed server-side call leaves `measureContext` null and every subsequent call throws again.

#### Build Output from `tsconfig.build.json`

`tsconfig.build.json` extends `tsconfig.json` and overrides:
- `noEmit: false` — enables output
- `allowImportingTsExtensions: false` — required for `.d.ts` emit (`.js` specifiers in source become valid imports)
- `types: []` — strips Bun/Node types from declaration output
- `rootDir: "./src"`, `outDir: "./dist"`
- `declaration: true` — emits `.d.ts` files
- `include: ["src/**/*.ts"]`, `exclude: ["src/layout.test.ts", "src/test-data.ts"]`

Result: `dist/layout.js` + `dist/layout.d.ts` (and all internal module files). The `package.json` `exports` map points the `"."` entry at `dist/layout.d.ts` (types) and `dist/layout.js` (runtime).

**Critical:** `dist/` is not checked into git. It must be generated by `bun run build:package` (`rm -rf dist && tsc -p tsconfig.build.json`) before the hub can resolve types via the package's `exports` map.

## Next.js Configuration

### `next.config.ts` for Submodule Consumption

Decision D-01 specifies path alias + `transpilePackages` + Turbopack `resolveAlias`. The hub lives at `hub/`, the submodule at `../pretext/` (relative to `hub/`).

```ts
// hub/next.config.ts
import type { NextConfig } from 'next'
import path from 'path'

const nextConfig: NextConfig = {
  transpilePackages: ['@chenglou/pretext'],
  experimental: {
    turbo: {
      resolveAlias: {
        '@pretext': path.resolve(__dirname, '../pretext/src/layout.ts'),
      },
    },
  },
}

export default nextConfig
```

**Why both `transpilePackages` and `resolveAlias`:**
- `transpilePackages` tells Next.js/SWC to compile this package through the hub's build pipeline instead of treating it as pre-built ESM.
- Turbopack's `resolveAlias` maps the module specifier to the actual file. Without it, Turbopack cannot resolve the path alias to the submodule source.
- The webpack fallback (for builds that still use webpack) needs a separate `webpack(config)` block pointing `config.resolve.alias['@pretext']` at the same path. Both should be present for robustness.

```ts
// hub/tsconfig.json — paths for TypeScript type resolution
{
  "compilerOptions": {
    "paths": {
      "@pretext": ["../pretext/src/layout.ts"]
    },
    "exclude": ["../pretext/src"]
  }
}
```

**Note on D-02 (type boundary):** For type-checking and production, the hub's `tsconfig.json` `paths` should point at `../pretext/src/layout.ts` during dev (direct source, no build step needed). The `exclude` list must explicitly exclude `../pretext/src/**` to prevent Hub's TypeScript 5.x compiler from picking up TS 6-specific flags (`allowImportingTsExtensions`, `verbatimModuleSyntax`, etc.) from Pretext's `tsconfig.json`.

### Port Configuration

Hub dev server must use port 3001. Pretext dev server claims port 3000 and kills whatever is on it (`lsof -tiTCP:$PORT -sTCP:LISTEN | xargs kill`). Configure in `hub/package.json`:

```json
"scripts": {
  "dev": "next dev --port 3001"
}
```

## React Hook Patterns

### `usePrepared(text, font)` — PTXT-01

```ts
// hub/src/lib/pretext/usePrepared.ts
'use client'
import { useEffect, useState, useRef } from 'react'
import { prepare, type PreparedText } from '@pretext'

// Module-level cache: survives component unmount/remount without re-preparing
// Key: `${font}::${text}` — font first so font changes invalidate cleanly
const preparedCache = new Map<string, PreparedText>()

export function usePrepared(text: string, font: string): { prepared: PreparedText | null; isLoading: boolean } {
  const [prepared, setPrepared] = useState<PreparedText | null>(null)
  const [isLoading, setIsLoading] = useState(true)

  useEffect(() => {
    let cancelled = false

    async function run() {
      // Critical: await font loading before prepare() or canvas measures fallback font
      await document.fonts.ready

      if (cancelled) return

      const key = `${font}::${text}`
      let hit = preparedCache.get(key)
      if (!hit) {
        hit = prepare(text, font)
        preparedCache.set(key, hit)
      }

      if (!cancelled) {
        setPrepared(hit)
        setIsLoading(false)
      }
    }

    setIsLoading(true)
    void run()
    return () => { cancelled = true }
  }, [text, font])

  return { prepared, isLoading }
}
```

**Key design choices:**
- Module-level `Map` (not component state or `useRef`) so the cache persists across remounts.
- `document.fonts.ready` guard inside the async effect — Inter may not be in the font cache on first cold load even with `next/font` preloading.
- `cancelled` flag prevents stale state updates after unmount or dep change.
- `prepare()` is NOT in a render loop — it only runs in `useEffect`.

### `useLayout(prepared, width, lineHeight)` — PTXT-02

```ts
// hub/src/lib/pretext/useLayout.ts
'use client'
import { useMemo } from 'react'
import { layout, type PreparedText, type LayoutResult } from '@pretext'

export function useLayout(
  prepared: PreparedText | null,
  width: number,
  lineHeight: number,
): LayoutResult | null {
  // layout() is pure arithmetic (~0.0002ms) — safe in useMemo
  return useMemo(() => {
    if (prepared === null || width <= 0) return null
    return layout(prepared, width, lineHeight)
  }, [prepared, width, lineHeight])
}
```

**Key design choices:**
- `layout()` is safe in `useMemo` — it has no side effects, no canvas calls, no DOM reads.
- Returns `null` while `prepared` is loading (matches `usePrepared`'s `isLoading` state).
- `width <= 0` guard prevents layout with zero container width on first render.
- Callers can also call `layout()` directly in `useLayoutEffect` for synchronous pre-paint measurement needed by virtualized lists.

### `useContainerWidth(ref)` — PTXT-03

```ts
// hub/src/lib/pretext/useContainerWidth.ts
'use client'
import { useEffect, useRef, useState } from 'react'

export function useContainerWidth(ref: React.RefObject<HTMLElement | null>): number {
  const [width, setWidth] = useState(0)
  const rafRef = useRef<number | null>(null)

  useEffect(() => {
    if (!ref.current) return

    // Initial measurement
    setWidth(ref.current.offsetWidth)

    const observer = new ResizeObserver(() => {
      // RAF debounce: coalesce burst resize events into one per frame
      if (rafRef.current !== null) cancelAnimationFrame(rafRef.current)
      rafRef.current = requestAnimationFrame(() => {
        if (ref.current) {
          setWidth(ref.current.offsetWidth)
        }
        rafRef.current = null
      })
    })

    observer.observe(ref.current)

    return () => {
      observer.disconnect()
      if (rafRef.current !== null) {
        cancelAnimationFrame(rafRef.current)
        rafRef.current = null
      }
    }
  }, [ref])

  return width
}
```

**Key design choices:**
- `requestAnimationFrame` debounce: one `setState` per frame at most, prevents excessive re-renders during smooth resize.
- Initial `setWidth` on mount so the first layout call has a real width, not 0.
- `rafRef` stored in `useRef` (not state) to avoid triggering re-renders when scheduling.

### `usePreparedWithSegments(text, font)` — PTXT-04

Same structure as `usePrepared` but calls `prepareWithSegments()` and returns `PreparedTextWithSegments | null`. Uses a separate module-level cache keyed by the same `${font}::${text}` pattern. The two caches (`preparedCache` and `preparedWithSegmentsCache`) are separate objects — a text can be in both if both hooks are used for the same string (rare but safe).

```ts
// hub/src/lib/pretext/fonts.ts — PTXT-06
// Single source of truth for the Pretext font string.
// Must exactly match the CSS font applied to text blocks.
// Never use system-ui, -apple-system, or CSS variables here.
export const PRETEXT_FONT = '16px Inter'
```

**Why `'16px Inter'` exactly:**
- Canvas `font` property uses the CSS font shorthand: `'<size> <family>'`
- The font family name must match what the browser resolved — `Inter` (capital I, as loaded by `next/font/google`)
- Size must be a pixel value matching the actual rendered font size in the DOM
- `system-ui` is forbidden: canvas and DOM resolve to different optical variants on macOS at sizes 10–12px, 14px, and 26px (documented in Pretext's `RESEARCH.md` and `CLAUDE.md`)

## Risks and Mitigations

### Risk 1: SSR throw from Pretext import leaking into Server Components
**Severity:** Critical — kills render with unhandled error  
**Trigger:** Any `import { prepare } from '@pretext'` reaching a Server Component, Route Handler, or `generateMetadata`.  
**Mitigation:**
- All files in `hub/src/lib/pretext/` marked `// CLIENT ONLY` at top of file.
- Every hook file starts with `'use client'` directive — this is enforced at the module level, not just the component level.
- Hub `eslint.config.js` can be configured to flag imports of `@pretext` outside of `'use client'`-marked files.
- In CI/dev: the sandbox page itself is the smoke test — if it builds without error, the SSR boundary is intact.

### Risk 2: `prepare()` called before fonts are loaded (font race condition)
**Severity:** High — produces silently wrong measurements cached permanently  
**Trigger:** `useEffect` fires before `document.fonts.ready` resolves on first cold page load  
**Mitigation:**
- `usePrepared` and `usePreparedWithSegments` both `await document.fonts.ready` before the first `prepare()` call.
- `next/font/google` with `display: 'swap'` preloads font files but does not guarantee the font is in the browser's font cache when the first effect fires.
- The `isLoading: true` state from `usePrepared` ensures the UI shows a skeleton until fonts are confirmed loaded and `prepare()` has run.
- **No fallback**: do not call `prepare()` speculatively before `fonts.ready` and re-prepare — `PreparedText` is opaque and there is no "reprepare" path. The measurement cache would need `clearCache()` + re-prepare, which is visible as a flicker.

### Risk 3: TypeScript 5.x (hub) vs TS 6 (pretext) compilation boundary
**Severity:** Moderate — type errors at hub `tsc --noEmit` time  
**Trigger:** Hub `tsconfig.json` accidentally includes `../pretext/src/` in its compilation, exposing TS 6-only flags  
**Mitigation:**
- Hub `tsconfig.json` `exclude` must list `"../pretext/src"` explicitly.
- `paths` points at `../pretext/src/layout.ts` for runtime resolution (dev) but the hub type-checker sees `PreparedText`, `LayoutResult`, etc. through the type-inferred path — these are TS 5-compatible types.
- For production type-checking against `dist/layout.d.ts`: `paths` can be switched to `"../pretext/dist/layout.d.ts"` after `bun run build:package` runs. The `postinstall` hook in hub `package.json` enforces that `dist/` is present.

### Risk 4: Turbopack `resolveAlias` for a path outside project root
**Severity:** Moderate — module-not-found at dev time if misconfigured  
**Trigger:** Turbopack may restrict alias resolution to within the Next.js project root by default  
**Mitigation:**
- The `resolveAlias` in `experimental.turbo` takes an absolute path (via `path.resolve(__dirname, '../pretext/src/layout.ts')`). Absolute paths bypass the project root restriction.
- `transpilePackages: ['@chenglou/pretext']` tells Next.js to treat this as a source-compilable package — without it, Next.js may try to load the package's `dist/` via the `exports` map (which may not exist during dev if `bun run build:package` hasn't run).
- If Turbopack still fails to resolve, fallback: add a webpack alias block alongside the Turbopack config as belt-and-suspenders.

### Risk 5: Git submodule not initialized in Vercel CI
**Severity:** High — build fails with missing `pretext/` directory  
**Trigger:** Vercel clones repos without `--recurse-submodules`  
**Mitigation:**
- Build command explicitly runs `git submodule update --init --recursive` as the first step.
- The `postinstall` script in hub `package.json` checks for `../pretext/dist/layout.js` and exits non-zero if missing, surfacing a clear error message before Next.js compilation starts.
- Vercel's build environment has git available; submodule init from a build command is a supported pattern.

### Risk 6: `useEffect` vs `useLayoutEffect` for initial height
**Severity:** Low-moderate — scroll jump in virtualized lists on first render  
**Trigger:** Using `useEffect` (fires after paint) for initial `layout()` call  
**Mitigation:**
- For the sandbox page (Phase 1), `useEffect` is acceptable — there is no virtualized list, so a post-paint height settle is fine.
- For Phase 2 (virtualized lists), `useLayoutEffect` must be used for the first `layout()` call to prevent scroll-jump. This is documented but not a Phase 1 concern.
- React 19 Strict Mode double-invokes effects in dev. The module-level `preparedCache` handles this correctly — second call is a cache hit. Verify during sandbox testing.

### Risk 7: `width: 0` on initial render before ResizeObserver fires
**Severity:** Low — produces `layout()` call with `maxWidth: 0`, returning `lineCount: N` (all segments on their own line)  
**Mitigation:**
- `useContainerWidth` sets width from `offsetWidth` immediately on mount (before the ResizeObserver callback fires).
- `useLayout` guards `if (width <= 0) return null` to skip the spurious layout call.
- The skeleton is shown while `prepared === null || width === 0`, so no visible artifact.

### Risk 8: Dual package manager contamination
**Severity:** Low — developer experience issue, not production issue  
**Trigger:** `npm install` run from repo root sweeping up `pretext/`  
**Mitigation:**
- No `package.json` at repo root — prevents npm/bun workspace accidental sweep.
- `setup.sh` explicitly `cd pretext && bun install` and `cd hub && npm install` — never a root-level install.
- Document in `CLAUDE.md`: "Pretext uses bun, Hub uses npm. Never cross-install."
